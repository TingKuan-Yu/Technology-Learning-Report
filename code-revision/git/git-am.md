# git am --reject 遇到conflict 但 git am --3way --keep-cr　確成功.　可能是什麼原因?

這是一個在處理 Kernel Patch 時非常經典的現象。簡單來說，`--reject` 是**純文字的比對（Textual Diff）**，而 `--3way` 是**語意上的合併（Three-way Merge）**。

以下是為什麼 `git am --3way` 能成功，而 `--reject` 會失敗的幾個核心原因：

### 1. 內容偏移與上下文失效 (Context Mismatch)

- **`--reject` 的行為**：它像 `patch` 指令一樣，強行嘗試將 Diff 的內容套用到檔案中。如果 Patch 指定的上下文（Context，即目標行數的前後三行內容）在當前檔案中發生了變化（例如別人改了變數名稱，或多了一行空行），`--reject` 就會直接報錯並產生 `.rej` 檔案。
- **`--3way` 的行為**：它會根據 Patch 內的元數據（Metadata），找到這份 Patch 的「原始基準點（Base Commit）」。它會嘗試建立一個暫時的虛擬祖先，然後執行像 `git merge` 那樣的演算法。只要該區塊的邏輯沒有直接衝突，Git 能處理一定程度的內容偏移。

### 2. 空白字元與換行符號的干擾 (`--keep-cr`)

- 您在指令中加上了 **`--keep-cr`**，這是一個關鍵。
- 如果 Patch 是在 Windows 環境產生的（帶有 `\r\n`），而您的 Linux Kernel 環境是 `\n`，`git am --reject` 會因為每一行末尾的字元不匹配而判定失敗。
- `--3way` 結合 `git` 內部的內容過濾機制，能更好地處理這些不可見字元的差異。

### 3. 虛擬祖先 (Virtual Ancestor) 的力量

當您使用 `--3way` 時，Git 會執行以下動作：

1. 讀取 Patch 中的 `index` 資訊（例如 `index ba8eb7d..25f5d54`）。
2. 在您的本地倉庫中尋找 `ba8eb7d` 這個提交（或類似的檔案狀態）。
3. 利用這個「共同祖先」來判斷哪些改動是您的，哪些是 Patch 的。
4. 即使目標行數已經跑到了檔案的另一端，只要中間沒有人改動到相同的內容，`--3way` 就能成功自動合併。

### 4. 指令特性對照

| **特性**         | **git am --reject**               | **git am --3way**                       |
| ---------------- | --------------------------------- | --------------------------------------- |
| **運作原理**     | 像 `patch` 工具，嚴格匹配上下文。 | 像 `git merge`，基於三方合併演算法。    |
| **對位移容忍度** | 低（Context 必須完全一致）。      | 高（只要內容邏輯不衝突）。              |
| **失敗結果**     | 產生 `.rej` 檔案，需手動修補。    | 進入衝突狀態，需手動 `git add` 後繼續。 |
| **依賴性**       | 僅依賴 Patch 文字內容。           | 依賴 Git 歷史中的 Index 資訊。          |

------

### 總結建議

在處理 Android GKI 或 Linux Kernel 的 Patch 時，**首選通常是 `--3way`**。

之所以會成功，是因為您的本地代碼雖然與 Patch 產生時的環境有些許不同（行數偏移或有無空行），但 Git 透過 **Index 資訊** 找到了它們的共同邏輯起點，並判定兩者的改動不互相排斥。

如果您想確認 Git 是如何做到的，可以在成功後執行：

Bash

```
git show --summary
```

檢查合併後的內容是否正確，特別是您提到的 `.setup` 或 `android_setup` 附近是否有重複或錯位的情況。
