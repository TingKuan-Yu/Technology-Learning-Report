## 在一個git branch中, 若中間有rebase等造成的分支, 那git log會將分支的log也印出嗎?

簡單來說：**會，Git 預設會印出所有「可達（Reachable）」的提交。**

在 Git 的邏輯中，`git log` 是沿著當前分支的節點（Commit）不斷往回溯源。如果你的分支歷史中包含 `rebase` 或 `merge` 造成的結構變化，它的顯示行為如下：

### 1. Rebase 的情況

`rebase` 的目的是「線性化」歷史。當你執行 `rebase` 時，Git 會把你分支上的提交「重新基底」到另一個分支的頂端。

- **結果：** `git log` 會看起來像一條直線。它會先顯示你新產生的提交，接著直接顯示你 rebase 對象（例如 `master`）的所有歷史。
- **現象：** 你不會看到「分支點」，但你會看到對象分支的所有舊 log 被串在你的紀錄之後。

### 2. Merge（合併）的情況

如果你使用的是 `git merge`，會產生一個「合併節點（Merge Commit）」。

- **結果：** `git log` 預設會進入分叉的分支，把該分支內所有的提交紀錄都抓出來，並按**時間順序**與主線的紀錄交錯排列。
- **現象：** 這就是為什麼有時候 `git log` 看起來很亂，因為它把側枝的詳細過程也印出來了。

------

### 如何控制顯示行為？

如果你只想看「主線」的邏輯，或者不想看到那些雜亂的分支紀錄，可以使用以下參數：

#### A. 只看一條主線：`--first-parent`

這是最有用的參數。它會讓 Git 在遇到合併節點時，只沿著「當前分支的第一個父節點」往回走，忽略被合併進來的分支詳細紀錄。

Bash

```
git log --first-parent --format="%h %ad %s" --date=short
```

#### B. 視覺化分支結構：`--graph`

如果你想看清楚到底哪裡分叉、哪裡合併，加上這個參數能畫出 ASCII 示意圖。

Bash

```
git log --graph --oneline
```

#### C. 限制不顯示特定分支的 log

如果你只想看 `branch_A` 有但 `master` 沒有的紀錄，可以使用 `..` 語法：

Bash

```
git log master..branch_A --oneline
```

### 總結

`git log` 本質上是**回溯歷史**。只要那個提交（即使是來自別人的分支）最終被併入了你的分支，它就是你歷史的一部分，`git log` 就會把它印出來。若要過濾掉，請使用 `--first-parent`。
