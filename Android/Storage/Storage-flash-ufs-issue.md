# 寫入放大（Write Amplification）是什麼?

在快閃記憶體（Flash Memory，如 SSD、eMMC、UFS、SD卡）的技術領域中，**寫入放大（Write Amplification, WA）** 是一個描述「實際寫入硬體的資料量」大於「主機端要求寫入的資料量」的現象。

這是面試時討論存儲效能與壽命時非常高階且核心的概念。以下我為您整理了詳細的解析報告。

### 給您的面試小提醒：

如果面試官問到 **寫入放大**，這通常代表他在考察你對 **FTL (Flash Translation Layer)** 的理解。

- **關鍵字：** 垃圾回收 (GC)、分區對齊 (Alignment)、抹除次數 (P/E Cycle)、F2FS。
- **具體經驗：** 提到你在做 Android 分區（Partitioning）時，會透過對齊底層硬體參數來解決這個問題，這會讓你聽起來像是一個非常有經驗的底層開發工程師。

## 深入理解寫入放大 (Write Amplification, WA)

### 1. 為什麼會產生寫入放大？

這源於快閃記憶體（NAND Flash）的物理特性限制：

- **讀取與寫入是以「頁 (Page)」為單位**（例如 8KB 或 16KB）。
- **抹除則是以「區塊 (Block)」為單位**（一個 Block 包含數百個 Pages）。
- **不能覆蓋寫入：** 當你要修改一個已經有資料的 Page 時，不能直接在原位修改（In-place update），必須先將整個 Block 抹除，或者將新資料寫到另一個空白的 Page。

### 2. 核心運作機制：垃圾回收 (Garbage Collection, GC)

當設備空間逐漸填滿，或是資料碎片化時，為了釋出空白的 Page，控制器（Controller）必須執行以下步驟：

1. 將某個 Block 中仍然「有效 (Valid)」的資料搬移到另一個新的 Block。
2. 將原本的 Block 整個抹除。
3. **搬移這些「有效資料」的操作，就是導致寫入放大的主因**，因為主機並沒有要求搬移這些資料，是控制器為了管理空間而產生的額外寫入。

### 3. 如何計算寫入放大率 (WAF)

寫入放大率（Write Amplification Factor, WAF）的公式如下：

$$WAF = \frac{\text{寫入快閃記憶體的總資料量}}{\text{主機端發出的寫入請求資料量}}$$

- **理想值：1.0**（寫入 1GB，硬體就只動 1GB）。
- **現實值：** 通常大於 1.0。若 WAF 為 5.0，代表你寫入 1GB 的檔案，硬體的壽命卻消耗了 5GB 的額度。

### 4. 寫入放大的影響

- **縮短壽命：** NAND Flash 的抹寫次數（P/E Cycles）有限，WA 越高，設備越快損壞。
- **降低效能：** 控制器忙著搬移內部資料，會佔用頻寬，導致主機端寫入速度變慢（常見的「掉速」現象）。
- **增加延遲：** 導致系統卡頓。

### 5. 如何優化並降低寫入放大？

在開發 Android 設備或嵌入式系統時，我們可以透過以下手段優化：

1. **分區對齊 (Partition Alignment)：**
   - 確保分區的起始位置是 **Erase Block Size** 的整數倍。如果沒對齊，一個文件系統的寫入動作可能會跨越兩個實體 Block，導致 WAF 翻倍。
2. **開啟 TRIM 命令 / Discard：**
   - **運作機制：** 在作業系統中刪除檔案時，文件系統僅會將該區域標記為「可寫入」，但底層的 Flash 控制器並不知道這些資料已經失效。
   - **技術細節：** 透過傳送 TRIM (SATA) 或 UNMAP (SCSI/UFS) 指令，主機端能主動告知控制器哪些 Page 已不再被使用。
   - **優點：** 控制器在進行「垃圾回收 (GC)」時，會直接跳過這些被標記的無效資料，無需搬移它們，從而顯著降低寫入放大並維持寫入速度。
   - **Android 實作：** Android 系統通常會透過 `fstrim` 服務（如 `vold` 定時觸發）或在 `fstab` 中掛載時加入 `discard` 選項來執行此操作。
3. **預留空間 (Over-Provisioning, OP)：**
   - 在分區規劃時預留一部分空間不給使用者，讓控制器有更多「緩衝區」來執行 GC，進而大幅降低 WAF。
4. **選擇適合的文件系統：**
   - 使用 F2FS（Flash-Friendly File System）取代傳統的 ext4。F2FS 採用 Log-structured 設計，能有效減少寫入放大。
5. **循序寫入優於隨機寫入：**
   - 儘量減少零碎的小檔案寫入，將資料整合成大塊後再下達寫入指令。



# 為什麼 Android 偏愛 F2FS，而 Ubuntu 堅持 ext4？

### 1. 核心設計哲學的差異

| 特性         | F2FS (Flash-Friendly File System)                            | ext4 (Fourth Extended Filesystem)           |
| ------------ | ------------------------------------------------------------ | ------------------------------------------- |
| **設計初衷** | 專為 NAND Flash 特性設計。                                   | 基於磁碟（HDD）演進，後來優化至 SSD。       |
| **寫入方式** | **LFS (Log-structured File System)**：將資料循序寫入新空間。 | **Overwrite**：傾向於在原位或附近修改資料。 |
| **主要優勢** | 降低寫入放大（WAF），大幅提升隨機寫入效能。                  | 極其穩定、成熟、修復工具極為完善。          |

### 2. 為什麼 Android (UFS) 需要 F2FS？

Android 設備（手機）的操作特性與 PC 非常不同：

- **高頻率隨機寫入：** 手機系統有大量的資料庫（SQLite）頻繁讀寫（如 LINE 訊息、系統 Log）。F2FS 的 LFS 架構能將隨機寫入轉化為循序寫入，這對 UFS 的效能提升非常有感。
- **儲存空間壓力：** 手機儲存空間通常較小且容易填滿，F2FS 在磁碟接近滿載時的效能下降比 ext4 緩慢。
- **硬體壽命：** 手機通常使用 2-4 年，降低寫入放大對保護不可更換的 UFS 顆粒至關重要。
- **現代手機配置：** 由於 Android 手機現在都有強大的電力管理與電池備援，F2FS 早期令人擔憂的「斷電資料遺失」風險在手機上相對較小。

### 3. 為什麼 Ubuntu (PC/Server) 仍堅持 ext4？

雖然 Ubuntu 也支援安裝在 F2FS 上，但 ext4 仍是預設首選，原因如下：

#### A. 穩定性與災難恢復 (Robustness)

- ext4 已經發展超過 15 年，經歷了無數次的壓力測試。在遭遇非預期斷電時，ext4 的日誌系統（Journaling）幾乎能保證文件系統結構不會崩壞。
- 對於 PC 使用者來說，資料的「絕對安全」高於那一點點感官上的讀寫速度提升。

#### B. 靜態資料與讀取優先

- PC 的使用情境中，作業系統檔案、遊戲、大型影片通常寫入後就很少變動（Read-heavy）。ext4 在處理這類靜態大檔案時的表現與 F2FS 差異極小。

#### C. F2FS 的缺點：碎片化與 CPU 開銷

- **碎片化 (Fragmentation)：** F2FS 運作久了，內部會產生嚴重的碎片化，需要複雜的 GC（垃圾回收）機制。這會消耗較多的 CPU 資源，在插電運行的 PC 上可能無感，但在伺服器環境中卻是不必要的變數。
- **GC 延遲：** F2FS 的 GC 有時會造成系統瞬間的卡頓，這對桌面系統的使用者體驗不見得是正向的。

#### D. 硬體層級的差異

- PC 使用的 **NVMe SSD** 通常擁有比手機 **UFS** 更強大的內部控制器（Controller）與更多的快取（DRAM Cache）。這代表 SSD 本身就能在硬體層級很好地處理寫入放大和負載平衡，不需要文件系統（F2FS）過度介入。

### 4. 總結對比

- **Android + F2FS = 性能導向：** 手機 UFS 控制器較弱，且隨機讀寫極多，F2FS 能補足硬體的短板，讓手機用久了也不容易變卡（解決「掉速」問題）。
- **Ubuntu + ext4 = 安全導向：** PC 的 SSD 硬體已經夠強，不需要犧牲穩定性去換取性能。ext4 的萬能與穩定，是 Ubuntu 作為通用作業系統的最佳保險。

### 5. 技術專家視角 (面試亮點)

如果您在面試中被問到此問題，可以這樣總結： 「F2FS 透過 **LFS 架構** 解決了 Flash 設備在**隨機寫入**上的物理痛點，非常適合手機這種 I/O 密集且硬體控制器相對簡化的環境。而 Ubuntu 選擇 ext4 是基於 **廣泛的相容性** 與 **成熟的數據恢復工具**；在擁有高效能控制器與獨立電源管理的 PC/Server 領域，ext4 的穩定性收益遠大於 F2FS 的性能收益。」
