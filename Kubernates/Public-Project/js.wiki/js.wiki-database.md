

# Why using database

這是一個非常精闢的問題！Wiki.js 的架構確實與許多傳統 Wiki 不同。

簡單來說：**Git 是你的「資料保險箱」（儲存端），而 Database (資料庫) 是你的「運作大腦」（執行端）。**

雖然 Wiki.js 支持將內容同步到 Git，但它**並非**直接從 Git 讀取檔案來渲染網頁。以下是為什麼它仍然需要資料庫（如 PostgreSQL）的幾個核心原因：

------

## 1. 效能與檢索 (Performance & Search)

Git 是一個版本控制系統，並不擅長快速查詢。

- **資料庫：** 當使用者訪問一個頁面時，Wiki.js 從資料庫讀取快取內容，速度是以毫秒計的。
- **Git：** 如果每次有人點擊網頁都要去掃描 Git Repo、解析 Markdown 檔案並轉換，伺服器負擔會非常重，且反應會很慢。
- **搜尋：** 資料庫負責處理**全文檢索和標籤**（Tags）篩選，這在純 Git 環境下很難高效達成。

## 2. 管理中繼資料 (Metadata Management)

許多 Wiki 的運作資訊並不適合存放在 Git 檔案中：

- **使用者權限：** 誰可以看哪個頁面？誰是管理員？這些權限邏輯存放在資料庫中。
- **資產管理：** 上傳的圖片、檔案的連結關係。
- **系統設定：** 你的外觀主題、郵件伺服器設定、認證模組（如 LDAP/SAML）配置。

## 3. 核心運作邏輯：同步（Syncing）

Wiki.js 採用的是一種**雙向同步機制**：

1. **寫入時：** 當你在 Wiki.js 編輯頁面並存檔，它會先寫入 **資料庫**，讓網頁立即更新。
2. **背景同步：** 隨後，Wiki.js 會啟動背景任務，將變更提交（Commit）到你的 **Git 儲存庫**。
3. **讀取時：** 平時都讀資料庫。只有當你手動觸發「從 Git 匯入」時，它才會掃描 Git Repo 並將內容更新回資料庫。

------

## 4. 總結：兩者的分工

| **元件**                     | **負責任務**                                     | **角色**                   |
| ---------------------------- | ------------------------------------------------ | -------------------------- |
| **Database** (PostgreSQL)    | 頁面即時渲染、使用者權限、搜尋索引、系統配置。   | **運作核心** (The Brain)   |
| **Git** (GitHub/GitLab/etc.) | 內容備份、版本歷史備援、外部編輯同步、災難還原。 | **持久儲存** (The Archive) |

### 這樣設計的好處？

即使你的 Wiki.js 伺服器壞了，只要 Git 還在，你的所有文章（Markdown）都安全無虞；反之，有了資料庫，你的 Wiki 才能像現代 Web App 一樣反應靈敏，並擁有複雜的權限管理。

> **延伸思考：** 如果你追求的是「完全不需要資料庫」的 Git Wiki，通常會選擇像是 **Obsidian + Quartz** 或 **Hugo/Jekyll** 這種「靜態網頁產生器」，但那就會失去 Wiki.js 提供的在線即時編輯與權限控管功能。
