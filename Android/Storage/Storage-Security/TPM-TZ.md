# TPM與Qualcomm TZ比較

在嵌入式系統與電腦架構中，**TPM (Trusted Platform Module)** 與 **Qualcomm TZ (TrustZone)** 都是為了實現「硬體信任根」（Root of Trust）而設計的安全技術，但它們的**設計哲學**與**運作方式**完全不同。

簡單來說：**TPM 是一個獨立的保險箱，而 TrustZone 是將房子隔出一個禁區。**

---

### 1. 核心定義與架構

#### **TPM (Trusted Platform Module)**

* **本質：** 是一個專用的**獨立安全微控制器**（通常是獨立的晶片，或整合在 SoC 內的硬體模組）。
* **設計目標：** 專注於儲存金鑰、執行加密運算以及「平台完整性度量」（測量 BIOS/核心是否被竄改）。
* **運作：** 它是被動的。主處理器發送命令給 TPM，TPM 處理完後回傳結果。

#### **Qualcomm TZ (TrustZone)**

* **本質：** 是 ARM 架構下的**系統級安全擴展**。它不是獨立晶片，而是將 CPU、記憶體和週邊設備劃分為兩個虛擬世界：**安全世界 (Secure World)** 與 **非安全世界 (Normal World)**。
* **設計目標：** 建立一個完整的「可信執行環境」（TEE），讓安全應用程式（如指紋辨識、支付邏輯）可以在隔離的環境中運行。
* **運作：** 它是主動的。CPU 可以在兩個世界間切換，安全世界擁有自己的作業系統（如 QTEE）。

---

### 2. 詳細對比

| 特性         | TPM (硬體或韌體式)                     | Qualcomm TrustZone (TEE)                        |
| ------------ | -------------------------------------- | ----------------------------------------------- |
| **隔離程度** | **物理隔離** (獨立電路/邏輯)。         | **邏輯隔離** (由 CPU 指令集與硬體防火牆隔離)。  |
| **運算能力** | 很弱，僅限於**簡單加密與簽章**。       | 很強，可以執行複雜的程式碼 (如臉部辨識演算法)。 |
| **存取資源** | 只能存取自己內部的暫存器。             | 可以直接存取受保護的螢幕、相機、指紋傳感器。    |
| **主要功能** | 金鑰保護、開機完整性檢測 (PCR)。       | 數位版權管理 (DRM)、行動支付、生物辨識。        |
| **典型應用** | Windows BitLocker 金鑰、企業筆電安全。 | Android 手機上的 Widevine L1、螢幕下指紋。      |

---

### 3. 在 Storage（儲存）部門的角度看兩者

如果你在面試 Storage 相關職位，主管可能會關注這兩者如何影響資料安全：

* **TPM 的作用：** 它通常用來存放**磁碟加密金鑰 (DEK)**。如果硬碟被拔到另一台機器，因為沒有原機器的 TPM 授權，數據就無法解鎖。它保護的是「靜態數據」。
* **TrustZone 的作用：** 它處理「運作中的安全」。例如，當你輸入加密磁碟的密碼時，密碼是直接輸入到 TrustZone 控制的受保護記憶體中，即使 Android 系統被植入木馬，也偷不到這組密碼。

---

### 4. 它們可以共存嗎？

**答案是：會的。** 在現代高階 Android 手機或 ARM 筆電中，通常會同時存在：

1. **TrustZone (TEE)** 提供一個安全的環境來執行指紋辨識程式。
2. **TPM (或類似 TPM 的安全元件/SE)** 用來儲存極其敏感的憑證或進行金融級的加密操作。

---

### 5. 總結

* 如果你需要一個**無法被讀取、絕對隔離**的小帳本，用 **TPM**。
* 如果你需要一個**受保護的私人辦公室**來處理複雜的敏感業務（如掃描人臉、解密影片），用 **Qualcomm TrustZone**。



# TZ 內security file system是如何運作?

**Security File System (SFS)** 在 **TrustZone (TZ)** 內的運作方式，他們通常是在對於「安全隔離」與「硬體信任鏈」的深度理解。

簡單來說，SFS 的核心目標是：**即使駭客取得了 Android (Normal World) 的最高權限（Root），他也無法讀取或竄改存放在 SFS 內的資料。**

以下是其運作的四大關鍵機制：

------

### 1. 虛擬層級的隔離 (The Virtual Layer)

SFS 並不是一個獨立的物理硬碟，它通常是存在於使用者資料分區（Userdata Partition）中的一個加密資料夾或特殊加密區塊。

- **Normal World (Android)：** 只看得到一堆亂碼檔案，無法解密，甚至無法直接存取該路徑。
- **Secure World (TZ/TEE)：** 透過 **Secure Storage 驅動程式**，將這些亂碼還原成可用的檔案系統。

------

### 2. 加密保護：綁定硬體的金鑰 (HUK)

SFS 內的所有資料都會經過加密，而加密金鑰的來源至關重要：

- **HUK (Hardware Unique Key)：** 這是燒錄在 SoC 硬體內、無法被讀出的金鑰。
- **金鑰衍生：** TZ 會利用 HUK 衍生出專屬於 SFS 的金鑰。這意味著**「這份資料離開這台手機就無法解密」**，即便把 Flash 晶片拆下來換到另一台手機也沒用。

------

### 3. 如何防止「重放攻擊」與「竄改」？

如果駭客無法解密 SFS，他可能會嘗試把舊的 SFS 備份蓋回去（重放攻擊）或是隨意修改幾個位元。SFS 透過以下技術防禦：

- **Merkle Tree (雜湊樹)：** SFS 會對所有檔案區塊建立雜湊樹。只要檔案被動過，樹根（Root Hash）就會不匹配。
- **RPMB (Replay Protected Memory Block)：** 這是 eMMC/UFS 內的一個特殊硬體區域。TZ 會將 SFS 的狀態（如寫入計數器或 Root Hash）存在 RPMB。因為 RPMB 強制要求身分驗證且防止重放，這保證了 SFS 的狀態永遠是最新的。

------

### 4. 運作流程範例：讀取一份安全憑證

1. **應用程式：** 向 Android 系統請求讀取金鑰（如指紋支付金鑰）。
2. **CA (Client App)：** 在 Android 層發送請求給 TZ 內的 **TA (Trusted App)**。
3. **TA：** 呼叫 TEE 作業系統提供的 SFS API。
4. **TEE OS：** 透過 UFS 驅動程式讀取加密的資料塊，並使用 HUK 衍生的金鑰進行解密。
5. **校驗：** 檢查 RPMB 中的數值，確保資料沒被「掉包」成舊版本。
6. **結果：** 資料在 TZ 內部處理完畢，僅將「驗證結果」傳回給 Android。

------

### 5. 面試官可能會追問：

- **效能損耗：** 「SFS 每次讀寫都要經過加密與雜湊驗證，這對大檔案處理有什麼影響？」
  - *回答思路：* 提到 **AES-XTS 模式** 與硬體加速引擎，以及 SFS 通常只存小體積敏感資料（金鑰、憑證），大檔案（如加密影片）會改用 **Inline Encryption**。

# 處理大檔案時，如何利用硬體加密引擎（Inline Encryption Engine）來減輕 CPU 在 TZ 或 TPM 之間傳輸數據的負擔嗎？

為了幫你把這系列關於 **Storage、UFS、安全性與效能** 的知識點串連起來，我整理了一個面試官可能會追問的「進階情境題」：

### 核心複習：當你在手機上寫入一個加密的大檔案時...

1. **檔案處理：** 你必須使用 **Chunking（分塊）**，避免 Android OS 因為記憶體不足而砍掉你的 Process。
2. **安全性（TZ vs TPM）：**
   - **TrustZone (TEE)** 會負責處理你的加密金鑰，確保金鑰不會暴露在非安全世界的 Android 核心中。
   - 如果你有硬體加密引擎（Inline Encryption），資料在流向儲存晶片的過程中會被實時加密。
3. **協定傳輸 (UPIU)：**
   - 加密後的資料會被打包成多個 **Data Out UPIU**。
   - 主機會透過 **Query Request UPIU** 檢查 UFS 的屬性（例如是否開啟了 Write Booster）。
4. **底層硬體 (JEDEC)：** 最終資料流遵循 **JEDEC** 定義的電氣標準，透過 M-PHY 物理層寫入快閃記憶體。
