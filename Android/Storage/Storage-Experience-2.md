# Android read only partition integrity是怎麼做的?

在 Android 系統中，確保「唯讀分區」（如 `/system`, `/vendor`, `/product`）不被竄改，是整個系統安全性的基石。這套機制主要由 **Verified Boot（驗證開機）** 結合 **dm-verity** 技術來實現。

這是一個由底層硬體到上層作業系統的「信任鏈」（Chain of Trust）。

------

### 1. 信任鏈的起點：Root of Trust

整個驗證過程是從硬體開始的：

1. **Bootloader 驗證：** 當你開機時，硬體唯讀記憶體（ROM）中的代碼會驗證 Bootloader 的數位簽章。
2. **Kernel 驗證：** Bootloader 接著驗證核心（Kernel）與 `vbmeta` 分區的簽章。`vbmeta` 包含了所有唯讀分區的驗證資訊。

------

### 2. 核心技術：dm-verity (Device Mapper Verity)

由於 `/system` 等分區檔案巨大（數 GB），系統不可能在開機時把整個分區讀完來檢查。因此，Android 使用了 Linux 核心的 **dm-verity** 功能。

#### **運作原理：哈希樹 (Hash Tree)**

- **分塊處理：** 唯讀分區被切分成無數個 4KB 的數據塊（Data Blocks）。
- **逐層雜湊：** 每個 4KB 塊都會產生一個雜湊值（Hash）。這些雜湊值會再兩兩成對產生上一層的雜湊，以此類推，最後形成一個 **Root Hash**。
- **簽章保護：** 這個 Root Hash 會被存放在受保護的 `vbmeta` 分區中，並由廠商的私鑰進行數位簽章。

------

### 3. 即時驗證：隨用隨查 (On-the-fly Verification)

這是最聰明的地方：**驗證發生在「讀取」的那一刻。**

1. 當系統需要讀取 `/system` 中的某個檔案時，`dm-verity` 驅動程式會攔截這個請求。
2. 它只讀取該檔案所在的 4KB 數據塊。
3. 它會即時計算該塊的雜湊，並與雜湊樹中對應的節點比對，直到回溯到已驗證過的 Root Hash。
4. **如果雜湊不匹配：** 系統會認為該區塊已被損壞或竄改，核心會立即採取行動（例如直接重啟手機，或讓該讀取請求失敗，導致 App 閃退）。

------

### 4. 針對 Android 15 與虛擬 A/B 的演進

由於你之前關心過 **Virtual A/B**，這對 Integrity 有額外的影響：

- **Snapshot 驗證：** 在進行 OTA 更新時，新版本的系統是以「快照（Snapshots）」形式存在的。Android 使用 **`dm-snapshot`** 與 `dm-verity` 配合，確保即使是在合併（Merge）前的虛擬分區，依然受到完整的雜湊樹保護。
- **FEC (Forward Error Correction)：** 為了提高穩定性，`dm-verity` 通常會搭配 **纠错碼（FEC）**。如果某個 4KB 區塊因為硬體老化損壞了一小部分，系統可以利用 FEC 自動修復，而不是直接讓手機無法開機。

------

### 5. 與 Storage 部門面試的連結

如果面試官問到這點，你可以展現更深層的思考：

- **I/O 效能：** 告訴面試官，雖然 `dm-verity` 增加了安全性，但也帶來了 CPU 運算開銷。現代 SoC 通常有硬體加速來處理 SHA-256 運算，以減少讀取延遲。
- **唯讀性：** 這也是為什麼你無法在不解鎖 Bootloader 的情況下，透過 `adb shell` 修改系統檔，因為即使你改了一個位元，雜湊樹校驗就會失敗，系統將無法進入。



# 再深入解釋 - Snapshot 驗證

這是一個非常深入且專業的領域。要理解 **Virtual A/B** 下的 Snapshot 驗證，我們得先釐清一個核心矛盾：`dm-verity` 要求資料是**唯讀且不可變**的，但 OTA 更新過程本質上是在**寫入**資料。

為了在更新過程中兼顧「安全性」與「空間效率」，Android 引入了一套複雜的堆疊架構。

------

### 1. 關鍵元件：`dm-snapshot` 與 `dm-user`

在虛擬 A/B 中，當系統處於更新後的「待合併」狀態時，你看到的「新系統分區」其實是由多層驅動程式疊加出來的虛擬視圖：

- **底層 (Base Device)：** 舊版本的實體分區（例如 Slot A 的資料）。
- **寫入層 (COW - Copy-on-Write)：** 存放新版本中「有變動」的數據塊。
- **合併層 (`dm-snapshot` 或 `dm-user`)：** 這是一個邏輯層，它負責判斷：如果你讀取的區塊在更新中沒變，就去讀底層；如果變了，就去讀 COW 層。

------

### 2. 如何在「拼湊」出的分區上做 Verity？

這就是最精妙的地方。Android 15 會為更新後的目標版本（Slot B）建立一套全新的 **Hash Tree**，並將這棵樹也存放在 COW 空間中。

驗證流程如下：

1. **請求讀取：** 系統請求讀取虛擬 Slot B 的某個 4KB 區塊。
2. **定位數據：** `dm-snapshot` 決定該區塊是來自「舊分區」還是「COW 空間」。
3. **提取數據：** 將數據塊提取到記憶體。
4. **校驗雜湊：** `dm-verity` 攔截這份數據，並去讀取預存在 COW 裡的 **新版 Hash Tree** 節點。
5. **比對：** 如果計算出的雜湊值符合新版的 Root Hash，數據才准許通過。

> **重點：** 即使數據散落在實體磁碟的不同角落（一部分在舊分區，一部分在虛擬快照中），`dm-verity` 看到的依然是一個完整的、邏輯上連續的鏡像。

------

### 3. Merging（合併）階段的驗證轉移

當你重啟進入新系統後，背景會開始進行 **Merge**（將 COW 的資料寫回實體 Super 分區）。這段時間最危險，因為資料正在移動。

- **雙重路徑驗證：** 在 Merge 過程中，如果系統需要讀取某個正在被移動的區塊，`snapuserd` (User-space snapshot daemon) 會確保 `dm-verity` 拿到的是正確的版本。
- **原子性保證：** 如果 Merge 到一半斷電，`dm-verity` 在下次開機時會偵測到狀態不一致，系統會退回到舊版本（Rollback），保證裝置不會變磚（Brick）。

------

### 4. 針對 Storage 主管面試的進階點：`dm-user`

在較新的 Android 版本（Android 12+）中，Google 傾向使用 **`dm-user`** 取代傳統的核心層 `dm-snapshot`。

- **為什麼要換？** 因為 `dm-user` 將快照的邏輯移到了 **User-space**（即 `snapuserd`）。
- **好處：** 這樣可以更靈活地支援 **壓縮（Compression）**。資料在 COW 中是壓縮過的，`snapuserd` 讀取後解壓，交給 `dm-verity` 驗證，最後才給系統使用。這大幅節省了手機儲存空間。

------

### 總結回答思路

如果被問到這個問題，你可以這樣總結：

> 「在 Virtual A/B OTA 期間，Android 透過 `dm-verity` 對邏輯上的快照裝置進行校驗。即使數據物理上分散在基底分區與 COW 區塊中，`dm-verity` 仍會利用儲存在快照元數據中的新版雜湊樹進行即時驗證。這確保了在數據合併（Merge）完成前，虛擬分區的完整性與原生分區一樣可靠。」

# 平衡 CPU 解壓縮效能與 I/O 驗證速度的權衡?

在 Android 的 Virtual A/B 架構中，這是一個極致的效能平衡遊戲。因為新版的系統資料在磁碟上是**壓縮過的（Compressed COW）**，而 `dm-verity` 驗證的是**解壓後的原始資料**。

這中間的權衡（Trade-off）主要圍繞著：**延遲（Latency）**、**吞吐量（Throughput）** 與 **CPU 負載**。

------

### 1. 核心矛盾：驗證與解壓的順序

在讀取一個區塊時，路徑如下：

```
磁碟讀取 (I/O)` → `解壓縮 (CPU)` → `雜湊計算 (dm-verity/CPU)` → `傳回系統
```

如果處理不好，這會造成明顯的系統卡頓。Android 透過以下幾個策略來優化：

#### **A. 選擇低運算成本的壓縮演算法**

Android 15 的虛擬 A/B 通常使用 **LZ4** 或 **ZSTD**：

- **LZ4：** 極快的解壓速度（數 GB/s），幾乎不增加延遲，但壓縮率較低。
- **ZSTD：** 壓縮率更好，但解壓時 CPU 負擔較重。
- **權衡：** 通常系統分區會優先選擇 LZ4，因為系統流暢度（低延遲）比省下幾百 MB 的空間更重要。

#### **B. 並行處理與 Pipeline（管線化）**

為了不讓 I/O 等待 CPU，`snapuserd` 採用了多執行緒架構。當 `dm-verity` 在驗證「區塊 A」的雜湊時，`snapuserd` 已經在背景預先從磁碟讀取並解壓「區塊 B」。

------

### 2. `snapuserd` 的優化技巧：快取與 Batching

為了平衡效能，`snapuserd` 引入了幾個關鍵機制：

- **Read-ahead (預讀)：** 既然 `dm-verity` 通常是循序檢查雜湊樹，`snapuserd` 會預測接下來要讀取的區塊，提早將壓縮塊讀入記憶體並解壓。
- **Worker Threads (工作執行緒)：** `snapuserd` 會根據 CPU 核心數開啟多個 Worker。如果你的手機是高階 SoC（如 Snapdragon 8 系列），它能同時解壓多個區塊，大幅抵銷 `dm-verity` 帶來的額外延遲。

------

### 3. 效能與空間的權衡表 (Trade-off Matrix)

| **優化方向**      | **採取手段**           | **優點**                      | **缺點 (權衡點)**                              |
| ----------------- | ---------------------- | ----------------------------- | ---------------------------------------------- |
| **追求極速 I/O**  | 關閉壓縮或用 LZ4       | 解壓極快，系統反應靈敏。      | COW 佔用 `/userdata` 空間巨大。                |
| **追求空間節省**  | 使用 ZSTD (高壓縮比)   | 適合小容量手機 (128GB 以下)。 | 更新後的 Merge 階段，手機會發熱且較耗電。      |
| **減少 CPU 負載** | 降低 FEC (糾錯碼) 比例 | 減少 `dm-verity` 的運算量。   | 磁碟若有微小損壞，系統會直接崩潰無法自我修復。 |

------

### 4. 針對 Storage 部門主管的進階回答

如果面試官問到：「如何進一步降低大檔案讀取的延遲？」你可以提到以下技術：

- **Direct I/O：** 在某些情況下繞過 Page Cache，減少記憶體拷貝次數。
- **Fixed-block Compression：** 確保壓縮塊與檔案系統的 4KB 區塊對齊。如果壓縮塊跨越了兩個物理區塊，解壓時會導致兩次 I/O，這會讓 `dm-verity` 的驗證速度減半。

------

### 5. 總結

在 Android 15 中，平衡點在於**「空間換取流暢度」**。雖然使用了虛擬 A/B 壓縮，但透過 `snapuserd` 在用戶空間（User-space）的高效率異步處理，加上 LZ4 等快速演算法，讓 `dm-verity` 的即時驗證幾乎感覺不到延遲。

# Direct I/O

在儲存系統與作業系統的效能優化中，**Direct I/O (O_DIRECT)** 是一個非常關鍵的技術。簡單來說，它允許**應用**程式**繞過核心（Kernel）的 Page Cache（頁面緩存）**，直接與儲存設備進行數據交換。

這對於處理「很大很大的檔案」（大數據、資料庫、大檔案串流）尤其重要。

------

### 1. 傳統 Buffered I/O vs. Direct I/O

#### **Buffered I/O (預設行為)**

當你讀取一個檔案時，核心會先將資料從磁碟拷貝到內部的 **Page Cache**，再從 Page Cache 拷貝到你的應用程式記憶體（User Buffer）。

- **優點：** 如果資料被反覆讀取，速度極快（直接從 RAM 讀取）。
- **缺點：** 產生兩次記憶體拷貝（Double Buffering），浪費 CPU 與記憶體帶寬。

#### **Direct I/O**

數據直接從磁碟控制器傳輸到應用程式指定的記憶體位址（通常配合 DMA 技術）。

- **優點：** 零拷貝（Zero-copy）感官體驗，大幅減輕 CPU 負擔與記憶體壓力。
- **缺點：** 失去了作業系統自動預取（Read-ahead）的優勢，若存取模式不規律，效能反而會變差。

------

### 2. Direct I/O 的三個嚴格限制

在 C++ 中使用 `open()` 系統調用並加入 `O_DIRECT` 標誌時，你必須遵守以下硬性規定，否則 I/O 會直接失敗：

1. **記憶體對齊 (Alignment)：** 應用程式提供的 Buffer 起始位址必須是磁區大小（通常是 512B 或 4KB）的整數倍。
2. **檔案位移對齊 (Offset)：** 讀寫的起始位置必須對齊磁區。
3. **長度對齊 (Length)：** 讀寫的資料長度必須是磁區大小的整數倍。

> **面試加分點：** 在 C++ 中，可以使用 `posix_memalign()` 來分配符合對齊要求的記憶體空間。

------

### 3. 為什麼 Storage 部門主管在意 Direct I/O？

當檔案「很大很大」時（例如 100GB），傳統的 Buffered I/O 會帶來嚴重的副作用：

- **Cache Thrashing（緩存抖動）：** 大檔案會迅速填滿 Page Cache，把其他重要的熱門資料（如系統常用的小檔案）踢出去。這會導致整個作業系統變得很卡。
- **控制權轉移：** 使用 Direct I/O，應用程式可以**自己管理緩存策略**。這在資料庫系統（如 MySQL, RocksDB）中非常常見，因為它們比 OS 更清楚哪些資料該留著。

------

### 4. 結合你之前的問題：Android 15 與 Direct I/O

在 Android 的 **Virtual A/B 更新**中，`snapuserd` 在將資料從 COW 合併（Merge）到實體分區時，往往會考慮使用 Direct I/O：

1. **效能：** 合併大分區是循序寫入，不需要經過 Page Cache 二次拷貝。
2. **記憶體節省：** OTA 更新通常在背景發生，不希望因為大檔案寫入佔用太多 RAM，影響使用者玩遊戲或滑社群媒體。

------

### 總結

- **Buffered I/O：** 適合小檔案、隨機讀寫。
- **Direct I/O：** 適合超大檔案、**順序讀寫**、或應用程式自帶緩存機制。

# F2FS vs EXT4 for userdata

在 Android 的 `userdata` 分區選擇上，這是一個關於**效能（F2FS）**與**成熟度（EXT4）**的經典權衡。身為 Storage 主管，面試官想看的是你是否理解檔案系統如何與底層 **NAND Flash** 特性（如寫入放大、垃圾回收）進行交互。

以下是針對 `userdata` 分區的深度對比分析：

------

### 1. 設計哲學的根本差異

- **EXT4 (Fourth Extended Filesystem):**
  - **原地更新 (Update-in-place)：** 當你修改檔案時，它會嘗試在磁碟的同一個位置覆寫數據。這是在 HDD 時代設計的邏輯。
  - **缺點：** 對於 Flash 而言，覆寫前必須先擦除（Erase-before-write），這會導致效能下降並加速硬體損耗。
- **F2FS (Flash-Friendly File System):**
  - **日誌結構 (Log-structured, LFS)：** 採用「追加寫入」模式，所有新數據都寫到**新的空白區塊**，而不是覆寫舊區塊。
  - **優點：** 完美契合 Flash 的物理特性，將大量的隨機寫入轉換為循序寫入。

------

### 2. 關鍵效能對比 (Performance)

| **特性**            | **EXT4**                     | **F2FS**                                    | **贏家** |
| ------------------- | ---------------------------- | ------------------------------------------- | -------- |
| **隨機讀寫 (IOPS)** | 較慢，尤其是當分區變髒時。   | **極快**，對於資料庫（如 SQLite）極其友善。 | **F2FS** |
| **抗碎片化**        | 較佳，結構簡單穩定。         | 長期使用後會碎片化，**需依賴 GC**。         | **EXT4** |
| **寫入放大 (WAF)**  | 較高，會產生較多無謂的擦除。 | **較低**，能有效延長 UFS 壽命。             | **F2FS** |
| **開機掛載速度**    | 極快且穩定。                 | 隨著檔案增多，掃描時間可能變長。            | **EXT4** |

------

### 3. F2FS 的殺手鐧：對抗「掉速」

手機用久了會卡頓，通常是因為隨機寫入效能下降。F2FS 引入了幾項針對 Android 優化的技術：

- **多日誌流 (Multi-stream)：** 將「冷數據」（如照片）與「熱數據」（如 App Log）分開存放，減少垃圾回收（GC）時需要搬移的資料量。
- **Inline Operations：** 對於極小的檔案，直接存在索引節點（Inode）中，減少 I/O 次數。
- **透明壓縮 (Transparent Compression)：** Android 11+ 的 F2FS 支援壓縮，這不僅節省空間，還因為減少了實際寫入閃存的數據量，反而提升了 I/O 壽命。

------

### 4. 為什麼 EXT4 仍未被淘汰？

儘管 F2FS 在效能上完勝，但很多中低階裝置仍選擇 EXT4：

1. **穩定性：** F2FS 的垃圾回收機制（GC）非常複雜，如果手機空間快滿了（>90%），GC 會為了尋找空間而瘋狂運作，導致手機發熱甚至卡死。
2. **記憶體開銷：** F2FS 運作時需要更多的系統記憶體（**RAM**）來維護其複雜的索引結構。
3. **掛載風險：** 在斷電、當機等異常情況下，EXT4 的日誌修復（fsck）比 F2FS 更可靠、更快速。

------

### 5. 專家面試點：原子寫入 (Atomic Writes)

如果面試官追問如何進一步提升資料庫效能，你可以提到：

> 「F2FS 支持 **Atomic Writes**。這讓 SQLite 這種資料庫在進行事務（Transaction）處理時，不需要再寫兩次數據（避免 Double Write Buffer），直接將寫入操作交給檔案系統保證原子性，這能讓資料庫效能翻倍。」

------

### 總結回答建議

> 「對於 `userdata` 這種充滿小檔案與頻繁資料庫操作的分區，**F2FS 是現代旗艦機的必然選擇**，它能顯著提升 App 啟動速度與系統流暢度。然而，在實作上，我們必須精確配置其 **背景 GC 策略**，並透過 Android 的 `vold` 服務定期觸發 `fstrim`，以確保在磁碟空間不足時，效能不會發生雪崩式的下滑。」

# F2FS 的 GC 運作邏輯（Foreground vs Background GC）

在面試中，提到 F2FS 就一定會被問到 **GC (Garbage Collection)**。身為 Storage 主管，他想知道你是否理解為什麼 F2FS 雖然快，但卻有「效能抖動（Jitter）」的風險。

F2FS 是**日誌結構檔案系統（LFS）**，它像是在紙上寫字，不擦掉舊的，而是直接找空白處寫新的。當空白處用完時，就必須進行 GC 來「搬移有效資料並擦除舊塊」。

F2FS 將此過程分為兩種模式：**Background GC**（背景回收）與 **Foreground GC**（前景回收）。

------

### 1. Background GC (BGGC) —「未雨綢繆」

這是最理想的狀態。當系統處於**閒置（Idle）**時，核心執行緒會悄悄啟動。

- **觸發時機：** 螢幕關閉、充電中，且系統偵測到 Free Segments 低於預設閾值。
- **運作邏輯：** * 它會挑選 **"Cold Segments"**（冷區塊，即資料很少變動的地方）進行搬移。
  - **貪婪策略 (Greedy Policy)：** 為了節省電力與 CPU，它會挑選有效資料最少的區塊。
- **優點：** 使用者完全無感，能確保隨時有乾淨的空白塊可用。

------

### 2. Foreground GC (FGGC) —「垂死掙扎」

這是 Storage 工程師最想避免的「噩夢模式」。

- **觸發時機：** 當磁碟空間極度不足（例如剩下不到 5%），且應用程式正嘗試**寫入資料**時。
- **運作邏輯：** * 因為已經沒有空白區塊可以寫入，寫入請求會被**阻塞（Block）**。
  - CPU 必須立刻停下手邊工作，強行搬移資料並回收空間，騰出位置後才允許應用程式繼續寫入。
- **代價：** * **極高的延遲：** 使用者會感覺手機明顯「卡死（Lag）」。
  - **寫入放大 (WAF)：** 因為是在壓力下運作，它不得不搬移一些原本不需要動的區塊。

------

### 3. GC 的挑選演算法：Cost-Benefit vs. Greedy

F2FS 在選擇哪個區塊（Segment）來回收時，有兩套智慧邏輯：

| **演算法**        | **適用模式** | **核心邏輯**                                                 |
| ----------------- | ------------ | ------------------------------------------------------------ |
| **Greedy (貪婪)** | **FGGC**     | 哪個區塊的「無效資料」最多就清哪個。目標是**最快**拿到空間，不計代價。 |
| **Cost-Benefit**  | **BGGC**     | 不只看有效資料量，還看**區塊的年齡（Age）**。如果一個區塊很久沒動了，回收它的效益最高，因為它變動機率小。 |

------

### 4. 面試官的高階追問：如何優化 GC？

如果你能在面試中主動提到以下優化，主管會覺得你非常有經驗：

- **SSR (Sticky Segment Reuse)：** 這是 F2FS 的一個絕招。當空間不足時，與其花大把力氣做 FGGC，不如找那些還有零星空位的區塊直接「塞進去」。雖然這會讓寫入變成隨機寫入，但比起讓系統卡死的 FGGC，這是一個很好的折衷。
- **IPU (In-place Update)：** 在特定壓力下，F2FS 會暫時放棄 LFS 模式，轉回像 EXT4 那樣原地覆寫，以避免觸發連鎖 GC。
- **FSTRIM / DISCARD：** 定期從 User-space 發送 TRIM 指令，告訴 F2FS 哪些資料已經被 App 刪除了，這能大幅減少 GC 需要搬移的「無效有效資料」。

------

### 5. 總結

- **BGGC** 是為了維持健康，利用零碎時間打掃。
- **FGGC** 是因為沒位置坐了，必須強行把人趕走，這會導致系統瞬間卡頓。

**這就是為什麼手機容量快滿時會變慢的原因。你想了解在 Android 程式開發中，有哪些 API 可以幫助我們偵測磁碟壓力，從而避免觸發 FGGC 嗎？**

# 比較UFS GC, F2FS GC,及Android GC

這是一個非常精彩的面試問題！這三個層級的 **GC (Garbage Collection)** 雖然都叫「垃圾回收」，但處理的對象、發生的層級以及對系統的影響完全不同。

將這三者放在一起比較，能體現出你對**「垂直整合儲存堆疊」**（Vertical Storage Stack）的深刻理解。

------

### 1. 三大 GC 定位對照表

| **特性**       | **Android GC (Java/ART)**          | **F2FS GC (File System)**     | **UFS GC (Firmware/FTL)**          |
| -------------- | ---------------------------------- | ----------------------------- | ---------------------------------- |
| **處理對象**   | 記憶體中的 **Object（物件）**      | 磁碟上的 **Segment（區塊）**  | 快閃記憶體中的 **Block/Page**      |
| **發生位置**   | RAM (系統記憶體)                   | Kernel Space (作業系統核心)   | UFS 控制器 (硬體韌體層)            |
| **回收目的**   | 釋放不再使用的變數空間。           | 整理檔案碎片，合併有效資料。  | 擦除無效塊，準備物理寫入空間。     |
| **對效能影響** | 造成 App 暫時停頓 (Jank/Stutter)。 | 造成磁碟 I/O 延遲，系統 Lag。 | 導致寫入放大，降低閃存壽命與速度。 |

------

### 2. 詳細運作邏輯分析

#### **(1) Android GC: 記憶體層級**

這是 **ART (Android Runtime)** 負責的。當你寫 Java/Kotlin 程式時，不再使用的物件會被回收。

- **機制：** 主要是標記-清除 (Mark-and-Sweep) 的變體。
- **面試關鍵：** 現代 Android 使用 **Concurrent Copying (CC) GC**，盡量在背景運行以減少「Stop-the-world」的時間，避免 UI 卡頓。

#### **(2) F2FS GC: 檔案系統層級**

這是作業系統為了對應 **LFS (Log-structured)** 寫入方式而做的整理。

- **機制：** 將多個零散且含有有效資料的 **Segments** 合併到一個新的 Segment，並將舊的標記為可釋放。
- **特殊點：** 它只知道哪些「檔案區塊」是無效的，但它無法直接操作硬體擦除。它必須透過 **TRIM/Discard** 指令告訴底層 UFS。

#### **(3) UFS GC: 硬體 FTL 層級**

這是 UFS 晶片內部的 **FTL (Flash Translation Layer)** 負責的。

- **機制：** NAND Flash 不能直接覆寫，必須先「擦除」才能「寫入」。UFS 必須不斷搬移數據以湊出完整的空白 Block。
- **挑戰：** UFS 其實不知道檔案系統裡哪些檔案被刪除了，除非 OS 發送了 **TRIM** 指令。否則，UFS 會徒勞無功地搬移那些「在 OS 層級已被刪除但硬體層級不知情」的廢物資料。

------

### 3. 三者的聯動效應（重要！）

在面試時，如果你能講出這段話，主管會覺得你是專家：

> 「這三個 GC 會產生**連鎖反應**。當 Android App 頻繁建立大量物件觸發 **Android GC**，會增加 CPU 負載；如果這些物件被寫入資料庫，會觸發 **F2FS 寫入**；當磁碟快滿時，**F2FS GC** 會開始搬移區塊，這會產生大量 I/O；最後，這些 I/O 傳送到硬體層，觸發 **UFS 內部的 GC**。如果三者同時在高負載下運作，就會造成嚴重的系統卡頓（Jank）與發熱。」

------

### 4. 如何優化這三種 GC？

- **優化 Android GC：** 減少記憶體抖動（Memory Churning），避免在 `onDraw()` 裡分配物件。
- **優化 F2FS GC：** 預留足夠的 **Over-provisioning (OP) 空間**。當剩餘空間充足時，GC 效率最高。
- **優化 UFS GC：** 定期執行 **Batch TRIM**。Android 系統通常在凌晨充電且閒置時，透過 `vold` 觸發 `fstrim`。

------

### 5. 主管可能會問的陷阱題：

**「為什麼有了 F2FS GC，底層 UFS 還是要做 GC？」**

- **標準答案：** 因為層級不同。F2FS 處理的是**邏輯位址 (LBA)** 的連續性，而 UFS 處理的是**物理位址 (PBA)** 的電學特性（擦除與電壓）。即便 F2FS 把資料排得很整齊，物理層可能因為多次寫入導致電子流失，仍需透過 **Wear Leveling** 或物理 GC 來維持數據可靠性。

# 檔案系統碎片化（Fragmentation）?

在面試中，如果你提到 EXT4，面試官可能會認為你偏向保守穩定。如果你能主動說出 EXT4 的**進階掛載參數（Mount Options）**，證明你能在穩定中壓榨出極限效能，會讓主管印象深刻。

針對寫入與讀取加速，EXT4 有幾個關鍵的設定值：

------

### 1. 最核心的加速：`data=ordered` vs `data=writeback`

這是 EXT4 對效能影響最大的設定。

- **`data=ordered` (預設)：** 先寫入數據，再更新檔案系統日誌（Metadata）。這是效能與安全的中間點。
- **`data=writeback`：** 不保證寫入順序， Metadata 更新與數據寫入異步進行。
  - **加速原理：** 減少了磁碟寫入的等待與同步次數。
  - **代價：** 系統當機時，新寫入的檔案可能會出現舊資料或亂碼。
  - **面試點：** 「在對一致性要求稍低的快取分區，我會選 `writeback` 來極大化寫入吞吐量。」

------

### 2. 減少 Metadata 寫入：`noatime` 或 `relatime`

這是最簡單也最有效的讀取優化。

- **`noatime`：** 讀取檔案時，不要更新檔案的「最後存取時間」。
  - **加速原理：** 在 Linux 中，即使只是「讀取」也會觸發一次「寫入」（更新時間戳）。開啟此項可以完全消除讀取過程中的額外寫入負擔。
- **`nodiratime`：** 同上，但針對目錄。

------

### 3. 優化延遲寫入：`commit`

- **`commit=N`：** 設定檔案系統每隔 `N` 秒才將 Metadata 與數據同步到磁碟。
  - **加速原理：** EXT4 預設是 5 秒。如果調大到 `commit=30` 或 `60`，可以讓多個小寫入在 Page Cache 中合併，減少磁碟 I/O 頻率。
  - **風險：** 斷電時會損失過去 `N` 秒內的資料。

------

### 4. 針對 Flash 設備的優化：`discard`

- **`discard` / `nodiscard`：**
  - 開啟 `discard` 會讓系統在刪除檔案時，立即對底層發送 **TRIM** 命令。
  - **效能權衡：** 即時 TRIM 可能會造成 I/O 抖動（Jitter）。在 Android 中，通常建議設為 **`nodiscard`**，並由系統在閒置時手動觸發批次 TRIM（`fstrim`），這對維持系統流暢度更有幫助。

------

### 5. 緩衝區與日誌優化：`barrier`

- **`barrier=0`：** 關閉寫入屏障（Write Barrier）。
  - **加速原理：** 告訴磁碟不用等待所有快取都確實落地，直接回報寫入完成。
  - **警告：** **這非常危險**，僅在硬體帶有 BBU（電池備援緩存）的 RAID 卡或高端手機端保護電路時才建議開啟，否則斷電可能導致檔案系統結構損壞。

------

### 綜合面試回答範例 (Storage 部門)

> 「為了優化 EXT4 的 `userdata` 效能，我會優先開啟 **`noatime`** 以減少不必要的 Metadata 寫入。在安全電路有保障的前提下，我會調整 **`commit=30`** 來合併 I/O。針對寫入密集型的非關鍵分區，我會考慮使用 **`data=writeback`** 並關閉 **`barrier`** 以壓榨出最高吞吐量。此外，我傾向在掛載時關閉即時 **`discard`**，改由背景服務定期進行批次 TRIM，以避免即時刪除檔案時造成的 UI 卡頓。」

### 總結設定表

| **參數**             | **作用**         | **推薦情境**                     |
| -------------------- | ---------------- | -------------------------------- |
| **`noatime`**        | 讀取不更新時間戳 | **所有情境建議開啟**             |
| **`data=writeback`** | 異步日誌寫入     | 快取分區 (Cache Partition)       |
| **`commit=60`**      | 延遲同步         | 大檔案循序寫入                   |
| **`barrier=0`**      | 關閉寫入屏障     | 極限效能測試 / 有 UPS 保護的硬體 |
| **`delalloc`**       | 延遲分配空間     | 預設開啟，有效減少碎片           |

# 重放攻击

**重放攻擊（Replay Attack）** 是一種網路攻擊手段，攻擊者攔截並記錄通訊雙方之間的合法數據（例如登入請求、交易指令），然後在稍後的**時間點將這些數據「原封不動地」重新發送給接收者**，以欺騙接收者執行未經授權的操作。

簡單來說，攻擊者不需要破解密碼或加密，只要**「錄下來，再播一次」**。

------

### 1. 舉個生活中的例子

想像一個遙控車鎖系統：

1. 你按下遙控器開車門，遙控器發射了一個加密訊號給車子。
2. 攻擊者躲在旁邊，用設備**錄下了**這個無線電訊號。
3. 等你離開後，攻擊者在車子旁邊**重放（播放）**這段訊號。
4. 車子以為是你又按了一次遙控器，於是門就開了。

------

### 2. 在電腦系統中的常見場景

- **身分驗證：** 攻擊者截獲你的登入封包，雖然內容加密了，但他直接把整個封包重發給伺服器，伺服器可能因此判斷他已通過驗證。
- **金融交易：** 攻擊者截獲你轉帳 100 元給他的請求，然後連續重發 10 次，如果你沒有防範重放機制，你帳戶裡就會被扣 1000 元。
- **會話劫持（Session Hijacking）：** 截獲有效的 Session Cookie 並重放，以獲取用戶登錄狀態。

------

### 3. 如何防禦重放攻擊？

針對這類攻擊，開發者通常會使用以下幾種技術：

| **防禦手段**                    | **原理**                                                     |
| ------------------------------- | ------------------------------------------------------------ |
| **時間戳 (Timestamp)**          | 每個請求帶上目前時間。接收端檢查如果請求時間與現在相差超過（例如 5 秒），就視為無效。 |
| **流水號/一次性隨機數 (Nonce)** | 每次請求生成一個唯一的隨機數。伺服器紀錄已用過的 Nonce，若收到重複的就拒絕。 |
| **序列號 (Sequence Number)**    | 在通訊過程中，給每個封包編號（1, 2, 3...）。如果收到重複或順序錯誤的編號，則丟棄。 |
| **一次性密碼 (OTP)**            | 像是 Google Authenticator，密碼每 30 秒變一次，即使截獲了，過期也無法再用。 |

------

### 4. 與 Storage 部門主管面試的連結

如果你是在應徵 **Storage 部門**，主管問這個可能與以下技術有關：

- **NVMe-oF (NVMe over Fabrics):** 在網路傳輸儲存指令時，如何確保指令不被惡意重放導致資料被反覆覆蓋？
- **加密磁碟 (SED):** 硬體加密金鑰在交換過程中，是否具備防止重放的握手機制（如使用的通訊協議是否包含 Sequence Number）。
- **檔案系統日誌 (Journaling):** 雖然不是攻擊，但系統崩潰後的「重做（Redo）」操作與重放邏輯相似，如何保證冪等性（Idempotency，即操作多次結果相同）也是關鍵。

# Android 将 userdata（/data 分区）设为小于 UFS/MMC 剩余物理空间

 Android 将 userdata（/data 分区）设为小于 UFS/MMC 剩余物理空间，是**多层级的系统设计与硬件特性适配**，核心目的是保障系统稳定、闪存寿命、数据安全与 OTA 升级可靠，而非单纯“浪费空间”。以下从底层硬件到上层系统，拆解背后的关键讲究与实操细节。

---

### 一、核心原因与设计目的（按优先级排序）

#### 1. 闪存硬件与分区表的强制预留（底层基础）

UFS/MMC 设备本身存在**物理坏块管理、OP（Over-Provisioning，超额配置）空间**，这部分空间不纳入用户可用分区：

- **OP 空间**：UFS/MMC 出厂时会预留 7–20% 的物理空间（容量越大，比例通常越低），用于坏块替换、GC 优化、TRIM 高效执行，直接降低写入放大（WAF）、延长闪存寿命，这部分空间对操作系统不可见，不属于“可用剩余空间”。
- **分区表与边界对齐**：Android 分区需按 UFS/MMC 的 4KB 最小可寻址块对齐，且要为 boot、system、recovery、vendor、odm 等系统分区预留固定空间，userdata 只能用“总容量 - 系统分区 - OP 空间”的剩余部分，而非“全部物理空间”。

#### 2. 文件系统级预留（系统维护必需）

userdata 常用的 ext4/F2FS 会预留空间，避免磁盘满导致系统崩溃：

- ext4 默认预留 **5% 空间给 root 用户**（可通过 tune2fs 调整），用于文件系统修复、日志写入、防止普通用户耗尽空间引发 OOM/进程崩溃。
- F2FS 会预留少量空间用于 GC 与元数据管理，提升随机写入性能与稳定性。
- 这部分预留对普通应用不可见，**表现为“userdata 可用空间 < 分区大小”**。

#### 3. 系统功能与升级的关键保障

- **OTA 升级预留**：userdata 需预留数百 MB 到数 GB 空间，用于存放 OTA 增量包、临时解压文件与升级过程中的数据迁移，避免升级失败（尤其是 A/B 分区升级时，需保证足够缓存空间）。
- **数据安全与恢复**：预留空间可用于系统恢复模式（recovery）的临时操作、日志存储、崩溃时的核心数据备份，降低数据丢失风险。
- **厂商定制预留**：厂商可能额外预留空间给系统应用、预装软件、主题/字体资源，或用于“安全分区”“隐私空间”等功能。

#### 4. 镜像烧录与动态扩容的兼容处理

- 量产烧录时，userdata 常通过固定大小的 userdata.img 写入分区，镜像大小由 BoardConfig.mk 中的 BOARD_USERDATAIMAGE_PARTITION_SIZE 定义，可能小于实际物理分区，恢复出厂设置会重新格式化并适配真实分区大小。
- 部分设备支持 GROW_LAST_PARTITION_TO_FILL_DISK 宏，让 userdata 动态扩容到剩余空间，但仍会保留文件系统与 OTA 所需预留。

#### 5. 性能与寿命优化

- 避免 userdata 空间 100% 占满：闪存满容量时 GC 效率骤降、写入放大飙升，预留空间可保证 GC 有足够空闲块，维持稳定读写性能与闪存寿命。
- 适配 UFS 的 SLC 缓存机制：预留空间可让控制器更高效地使用 SLC 缓存，提升小文件写入性能。

### 二、关键概念区分（避免混淆）

| 空间类型              | 定义                                    | 大小范围              | 是否对用户可见               |
| :-------------------- | :-------------------------------------- | :-------------------- | :--------------------------- |
| **物理总容量**        | UFS/MMC 标称容量（如 256GB）            | 标称值                | 否（含 OP 与坏块空间）       |
| **可用物理空间**      | 总容量 - OP - 坏块                      | 标称的 80–93%         | 否（系统分区占用后剩余）     |
| **userdata 分区大小** | 系统分区外分配给 /data 的空间           | 可用物理空间的 80–95% | 部分可见（通过 df 查看）     |
| **用户可用空间**      | userdata 分区 - 文件系统预留 - 系统占用 | 分区大小的 95–98%     | 是（设置中显示的“内部存储”） |

### 三、Linux/Android 实操：查看与调整预留空间

#### 1. 查看各层级空间

```bash
# 1. 查看 UFS/MMC 物理容量与 OP 空间（root）
cat /sys/class/ufs/ufs0/geometry_descriptor  # UFS
cat /sys/class/mmc_host/mmc0/mmc0:0001/ext_csd  # eMMC

# 2. 查看分区表与 userdata 分区大小
lsblk -o NAME,SIZE,TYPE,MOUNTPOINT /dev/ufs0  # 或 mmcblk0

# 3. 查看文件系统预留空间（ext4 示例）
tune2fs -l /dev/block/by-name/userdata | grep "Reserved block count"
```

#### 2. 调整文件系统预留（谨慎操作）

```bash
# 降低 ext4 预留空间到 1%（提升用户可用空间，适合大存储设备）
tune2fs -m 1 /dev/block/by-name/userdata
# 恢复默认 5%
tune2fs -m 5 /dev/block/by-name/userdata
```

#### 3. 解决镜像与物理分区不匹配

恢复出厂设置可重新格式化 userdata，适配真实物理分区大小，替代烧录时的固定镜像大小限制。

### 四、最佳实践与风险提示

1. **保留默认预留**：普通设备不建议关闭/大幅降低预留，否则可能导致 OTA 失败、系统崩溃、闪存寿命缩短。
2. **大存储设备优化**：256GB+ UFS 可将 ext4 预留降至 1–2%，平衡可用空间与稳定性。
3. **避免 userdata 100% 占用**：通过监控工具（如 df、proc/meminfo）确保 userdata 使用率低于 90%，防止性能下降。
